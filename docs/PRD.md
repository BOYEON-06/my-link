# 마이링크 (MyLink) 요구사항 정의서 (PRD)

## 1. 프로젝트 개요
* **프로젝트명:** 마이링크 (MyLink)
* **서비스 목적:** 개발자 및 크리에이터가 자신의 다양한 포트폴리오, 깃허브(Github), 소셜 미디어, 개별 프로젝트 링크를 하나의 페이지로 모아 효과적으로 공유하고 관리할 수 있는 링크 집합(Link-in-Bio) 서비스.
* **대상 사용자:** 자신의 작업물과 활동 내역을 하나의 프로필로 알리고자 하는 개발자, 디자이너, 콘텐츠 크리에이터 등.
* **비즈니스 모델:** 전면 무료 서비스 (수익화 및 유료 프리미엄 모델 제외)

## 2. 사용 기술 및 공통 사항 (Overview)
* **디자인 및 UI:** 추후 도입될 별도의 **디자인 시스템**을 전체적으로 적용 예정.
* **인증 방식:** 구글 소셜 로그인만 지원 (**Firebase Authentication** 적용).
* **데이터베이스 모델링:** 사용자 정보에는 `displayName` 필드를 필수로 포함하며, 개별 사용자의 링크 등 세부 데이터는 **서브 컬렉션(Subcollection)**으로 구조화하여 관리한다.
* **기기 지원:** 모바일 중심(Mobile-First) 디자인을 따르되, 데스크탑 및 태블릿 환경에 완벽하게 대응하는 반응형 웹.

## 4. 각 기능 상세 설명

### 4.1 로그인 및 회원가입 (Auth)
* **Firebase 기반 구글 로그인:** 별도의 아이디/비밀번호 입력 과정 없이 구글 계정을 통해서만 빠르고 안전하게 가입 및 로그인을 처리한다. 비밀번호 찾기, 이메일 인증, 이메일/비밀번호 변경 기능은 구현하지 않는다.

### 4.2 프로필 설정 및 도메인 생성
* **고유 Username 설정:** 가입 후 첫 화면에서 영문/숫자 기반의 고유한 Username을 설정한다. 이는 곧 본인의 프로필 페이지 URL 주소가 되며, 사용자 간 중복이 불가능하다.
* **비주얼 프로필 및 닉네임 수정:** 크리에이터 아이덴티티를 나타내는 프로필 이미지를 업로드하고, 데이터베이스 상의 `displayName`을 활용하여 활동명(닉네임)과 소개글(Bio)을 설정한다. **설정된 프로필 이미지, 닉네임 및 소개글은 언제든지 자유롭게 수정할 수 있다.**
* **인라인 편집 (Inline Editing):** 프로필과 닉네임 등의 정보를 수정할 때, 별도의 수정 페이지로 이동하지 않고 화면 내에서 텍스트를 바로 클릭하여 수정하는 직관적인 인라인 편집 방식을 사용한다.

### 4.3 메인 링크 데이터 및 관리 (CRUD & 정렬)
* **서브 컬렉션 기반 링크 블록 추가:** 가장 직관적인 형태로 URL과 타이틀을 입력하여 링크 버튼을 생성하며, 해당 데이터는 사용자 도큐먼트의 서브 컬렉션(Subcollection)에 구조화되어 저장된다.
* **링크 인라인 편집 및 파비콘 (Google API):** 생성된 링크 블록의 내용(타이틀, URL) 역시 글자를 클릭해 화면에서 즉시 변경하는 **인라인 편집**을 지원한다. 또한 URL을 입력하면 **Google Favicon API**를 호출하여 해당 도메인의 파비콘 이미지를 링크 블록 좌측에 자동으로 불러온다.
* **드래그 앤 드롭 정렬:** 관리자 페이지에서 링크 블록을 마우스 혹은 터치로 잡고 위아래로 드롭하여 노출 순서를 쉽게 정리할 수 있다.
* **토글 활성화:** 삭제하지 않고도 특정 링크의 공개/비공개 상태를 스위치(토글) 버튼으로 즉시 제어할 수 있다.

### 4.4 리치 미디어 링크 (크리에이터/개발자 특화)
* **미디어 임베딩:** 단순한 링크 이동 버튼에 그치지 않고, 유튜브(YouTube) 영상 썸네일과 플레이어, 깃허브 등 관련 컴포넌트를 링크 블록에 표출한다.

### 4.5 마케팅적 보조 기능 (하이라이트 및 스케줄링)
* **하이라이트 애니메이션:** 사용자가 가장 강조하고 싶은 링크에 통통 튀는 애니메이션이나 테두리 강조 효과를 부여한다.
* **링크 오픈 예약(기간 한정):** 특정 기간에만 프로필에 링크가 나타나게 자동화한다.

### 4.6 오프라인 친화성 (QR 코드)
* **QR 코드 렌더링:** 사용자의 프로필 URL로 바로 연결되는 QR 코드를 동적으로 발급하며, 컨퍼런스 명함이나 발표 자료의 레퍼런스로 손쉽게 첨부할 수 있도록 지원한다.

## 5. 데이터베이스 모델링 (Firebase Firestore)

Firebase Firestore(NoSQL)의 도큐먼트 모델을 기반으로 하며, 사용자 데이터와 다수의 링크 데이터를 효율적으로 관리하기 위해 **서브 컬렉션(Subcollection)** 구조를 활용한다.

### 5.1 콜렉션 스키마 명세

데이터 구조를 한눈에 파악할 수 있도록 타입스크립트(TypeScript) 인터페이스 형식의 코드 박스로 구조를 정의합니다.

```typescript
// 1) `users` 컬렉션 (사용자 기본 정보)
// Document Path: /users/{uid}
interface UserDocument {
  username: string;       // 고유 개인 URL 식별자 (필수, 중복 불가)
  displayName: string;    // 사용자 닉네임 (인라인 편집 가능)
  photoURL: string;       // 프로필 이미지 주소
  bio: string;            // 사용자 소개글
  createdAt: Timestamp;   // 데이터 생성 일시
}

// 2) `links` 서브 컬렉션 (링크 블록 리스트)
// Document Path: /users/{uid}/links/{linkId}
interface LinkDocument {
  title: string;          // 버튼에 표시될 링크 이름 (인라인 편집 가능)
  url: string;            // 이동할 실제 위도착지 주소
  faviconUrl: string;     // 구글 파비콘 API로 가져온 이미지 경로
  order: number;          // 화면에 표시될 정렬 순서값 (Drag & Drop 교체)
  isActive: boolean;      // 링크 공개/비공개 토글 여부
  isHighlighted: boolean; // 강조 애니메이션 효과 적용 여부
  scheduledStartsAt?: Timestamp; // [선택] 링크 활성화 예약 시작 시간
  scheduledEndsAt?: Timestamp;   // [선택] 링크 종료 예약 시간
  clickCount: number;     // [추후 V2] 개별 링크 클릭(조회)수 데이터
}
```
